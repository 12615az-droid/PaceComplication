АРХИТЕКТУРА ПРИЛОЖЕНИЯ
1 Общая схема
	Приложение построено с разделением ответственности между слоями.
	Каждый слой выполняет свою задачу и не вмешивается в логику других.
	Основные принципы:
		простота
		читаемость
		возможность расширения без переписывания ядра

	UI слой (Jetpack Compose)
	UI отвечает только за отображение данных и обработку действий пользователя.
	Бизнес-логика и расчёты в UI отсутствуют.

2 Особенности UI слоя:
	все данные приходят через StateFlow
	UI автоматически обновляется при изменении состояния
	UI не знает о сервисах и GPS напрямую

	Основные элементы UI:
		MainActivity
			точка входа приложения
			запрос разрешений
			запуск и остановка сервиса
		PaceScreen  
 			stateful-обёртка  
 			подписка на состояние из LocationRepository  

 		TrainingScreen  
 			чистый UI основного экрана  
 			отображение темпа и статуса  

 		SettingsScreen  
 			экран настроек  
 			находится в разработке  

 		HistoryScreen  
 			экран истории  
 			пока используется как заглушка  
3. Слой состояния и логики (LocationRepository)
	LocationRepository является центральным хранилищем состояния приложения.

	Назначение:
		хранение состояния трекинга
		расчёт темпа
		фильтрация GPS данных
		подготовка данных для UI и часов

	Почему используется singleton:
		одно состояние на весь процесс
		простой доступ из UI и сервиса
		отсутствие необходимости в DI

	Хранимое состояние (StateFlow):
		текущий темп
		точность GPS
		режим активности
		флаг активного трекинга

	Основная логика:
		фильтрация по точности GPS
		отбрасывание нефизичных скоростей
		EMA-сглаживание темпа
		форматирование темпа в мм:сс
		отправка данных на Wear OS
4 Системный слой (LocationService)
	LocationService отвечает за работу с системой Android.

	Назначение:
		получение GPS координат в фоне
		работа в режиме foreground service
		передача данных в LocationRepository

	Особенности:
		используется FusedLocationProviderClient
		обновления координат раз в секунду
		сервис не содержит UI логики
		ресурсы корректно освобождаются при остановке
5 Уведомления
	Работа с уведомлениями вынесена в отдельный класс.

	LocationNotificationHelper отвечает за:
		создание NotificationChannel
		сборку foreground уведомления
		управление MediaSession

	Особенности уведомления:
		уведомление нельзя смахнуть
		не воспроизводит звук при обновлении
		использует MediaStyle
		кнопки управления заложены, но пока заглушки
6 Разрешения и жизненный цикл
	Работа с разрешениями сосредоточена в MainActivity.

	Особенности:
		разрешения запрашиваются только при необходимости
		повторный запрос не выполняется, если разрешения уже выданы
		Activity остаётся тонкой и не содержит бизнес-логики
7 Wear OS
	Реализован обмен данными с часами.

	Используемая технология:
		Data Layer API
		DataClient

	Текущее состояние:
		на часы передаётся текущий темп
		отображается одно числовое значение
		обновление происходит с задержками
		интерфейс часов минимальный
8 Отладка и диагностика
	Для анализа работы приложения используется TelemetryLogger.

	Назначение:
		логирование темпа и точности
		фиксация состояния батареи
		анализ поведения GPS

	Особенности:
		вывод в Logcat
		запись в текстовый файл
		не влияет на бизнес-логику приложения
